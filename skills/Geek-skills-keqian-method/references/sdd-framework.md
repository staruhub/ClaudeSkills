# SDD文档驱动开发框架

## 核心理念

不写文档 → 没有架构观 → AI只能盲干 → 质量不可控

文档不是给人看的附属品，而是给AI看的"操作手册"。
文档质量直接决定AI产出质量。

---

## 文档层次结构

```
Level 0: 产品愿景文档（Why）
├── 产品定位、目标用户、核心价值
└── 人工撰写，AI辅助优化

Level 1: PRD / 需求文档（What）
├── 功能列表、用户故事、验收标准
├── AI辅助撰写 + 人工深度审核
└── 每个功能点至少5个用例（不是3个敷衍）

Level 2: 技术架构文档（How）
├── 技术栈选择（模板化决策矩阵）
├── 模块划分、接口规范、数据模型
├── AI维护 + 人工把关
└── 变更时自动触发文档更新

Level 3: 实现指南（How exactly）
├── 编码规范、命名约定、目录结构
├── 直接嵌入repo（AGENTS.md / CLAUDE.md）
└── Agent直接读取执行

Level 4: 质量定义（How good）
├── Eval标准、门禁规则、threshold定义
├── 每个功能对应的验收脚本
└── 自动化执行，人工仲裁
```

## 文档 vs 代码的精力分配

| 阶段 | 文档 | 代码 | 说明 |
|------|------|------|------|
| 项目启动 | 90% | 10% | 几乎全部精力在文档 |
| 核心开发 | 60% | 40% | 持续维护文档 |
| 迭代收敛 | 40% | 60% | 文档趋于稳定 |
| 长期维护 | 70% | 30% | 文档是harness的核心 |

总体平均：**70%文档 / 30%代码**

## 技术栈选择模板

技术栈选择是套路化的事，不需要每次重新思考：

```markdown
### 决策矩阵

| 维度 | 选项A | 选项B | 选项C | 权重 |
|------|-------|-------|-------|------|
| 团队熟悉度 | | | | 30% |
| 社区生态 | | | | 20% |
| AI Agent友好度 | | | | 25% |
| 长期维护成本 | | | | 15% |
| 性能需求匹配 | | | | 10% |

注意：AI Agent友好度是新增维度，包括：
- 模型训练数据中该技术栈的覆盖度
- 代码补全/生成的准确率
- 调试信息的可读性
```

## 文档自动维护机制

```
代码变更 PR
    ↓
门禁检查：文档是否需要同步更新？
    ↓ 需要
自动生成文档更新PR
    ↓
人工审核文档更新
    ↓
合并
```

关键：让AI自己维护文档，代码再vibe对齐。
不要让文档和代码脱节，这是文档驱动的死穴。

---

## 文档门禁规则

1. **新增功能**：必须先有文档，再写代码
2. **技术栈变更**：必须更新决策矩阵
3. **接口变更**：必须同步更新接口文档
4. **架构变更**：必须更新架构图和决策记录
5. **质量标准变更**：必须更新eval定义

门禁不通过 → 不允许合并代码。

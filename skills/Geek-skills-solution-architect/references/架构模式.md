# 软件架构模式详解

## 一、分层架构 (Layered Architecture)

### 概述
将系统划分为多个层次,每层有特定职责,上层可以调用下层,但下层不能调用上层。

### 典型分层
1. **表示层 (Presentation Layer)**: UI、API接口
2. **业务逻辑层 (Business Logic Layer)**: 业务规则、领域模型
3. **数据访问层 (Data Access Layer)**: 数据库操作
4. **数据库层 (Database Layer)**: 持久化存储

### 优点
- ✅ 结构清晰,易于理解
- ✅ 职责分离,易于维护
- ✅ 易于测试和替换层

### 缺点
- ❌ 性能开销(层间调用)
- ❌ 部署不灵活
- ❌ 扩展性受限

### 适用场景
- 传统企业应用
- 中小型系统
- 团队技能偏传统

### 示例
```
┌──────────────────┐
│   Presentation   │  (Web UI, API)
├──────────────────┤
│  Business Logic  │  (Services, Domain)
├──────────────────┤
│   Data Access    │  (Repositories, ORM)
├──────────────────┤
│    Database      │  (MySQL, PostgreSQL)
└──────────────────┘
```

## 二、微服务架构 (Microservices Architecture)

### 概述
将应用分解为小型、独立的服务,每个服务运行在自己的进程中,通过轻量级机制(通常是HTTP/gRPC)通信。

### 核心特征
- 服务独立部署
- 去中心化治理
- 独立数据存储
- 容错设计
- 自动化部署

### 优点
- ✅ 独立开发和部署
- ✅ 技术异构性
- ✅ 容错隔离
- ✅ 可独立扩展
- ✅ 团队自治

### 缺点
- ❌ 分布式系统复杂度
- ❌ 数据一致性挑战
- ❌ 服务间通信开销
- ❌ 运维复杂度高
- ❌ 测试困难

### 适用场景
- 大型复杂系统
- 多团队开发
- 不同服务有不同扩展需求
- 需要技术多样性

### 关键考虑
1. **服务划分**: 按业务能力或领域划分
2. **服务发现**: Consul, Eureka, Kubernetes Service
3. **API网关**: 统一入口、路由、认证
4. **配置管理**: 集中配置中心
5. **监控追踪**: 分布式追踪、日志聚合
6. **容错机制**: 熔断、降级、重试

### 示例架构
```
                ┌────────────┐
                │ API Gateway│
                └─────┬──────┘
         ┌────────────┼────────────┐
    ┌────▼────┐  ┌───▼────┐  ┌───▼────┐
    │User Svc │  │Order Svc│  │Pay Svc │
    │(Node.js)│  │(Java)   │  │(Go)    │
    └────┬────┘  └───┬────┘  └───┬────┘
    ┌────▼────┐  ┌───▼────┐  ┌───▼────┐
    │User DB  │  │Order DB│  │Pay DB  │
    └─────────┘  └────────┘  └────────┘
```

## 三、事件驱动架构 (Event-Driven Architecture)

### 概述
系统组件通过产生、检测和响应事件来进行通信和协调。

### 核心组件
- **事件生产者**: 生成事件的组件
- **事件消费者**: 处理事件的组件
- **事件总线/代理**: Kafka, RabbitMQ, AWS EventBridge

### 模式

#### 3.1 发布-订阅 (Pub/Sub)
```
Producer → Topic → [Subscriber1, Subscriber2, Subscriber3]
```
- 一对多通信
- 消费者解耦

#### 3.2 事件溯源 (Event Sourcing)
- 存储领域对象的所有状态变更
- 可重放事件历史
- 审计追踪完整

#### 3.3 CQRS (命令查询职责分离)
- 读写分离
- 写入用命令模型
- 查询用查询模型
- 常与事件溯源结合

### 优点
- ✅ 高度解耦
- ✅ 可扩展性强
- ✅ 实时响应
- ✅ 异步处理

### 缺点
- ❌ 复杂度高
- ❌ 调试困难
- ❌ 最终一致性
- ❌ 事件顺序问题

### 适用场景
- 实时数据处理
- IoT系统
- 金融交易
- 电商订单流程
- 微服务间通信

### 示例
```
┌──────────┐      Event      ┌──────────┐
│ Order    │──────────────▶ │  Kafka   │
│ Service  │  OrderCreated  └────┬─────┘
└──────────┘                     │
                         ┌───────┼───────┐
                    ┌────▼────┐  │  ┌───▼────────┐
                    │Inventory│  │  │Notification│
                    │ Service │  │  │  Service   │
                    └─────────┘  │  └────────────┘
                            ┌────▼────┐
                            │Payment  │
                            │Service  │
                            └─────────┘
```

## 四、无服务器架构 (Serverless Architecture)

### 概述
开发者只需要关注代码,无需管理服务器。云供应商自动处理扩展、容错等。

### 核心概念
- **FaaS** (Function as a Service): AWS Lambda, Azure Functions
- **BaaS** (Backend as a Service): Firebase, Auth0
- **按需付费**: 只为实际使用付费
- **自动扩展**: 从0到无限扩展

### 优点
- ✅ 零服务器管理
- ✅ 成本效益高(按使用付费)
- ✅ 自动扩展
- ✅ 快速开发和部署
- ✅ 内置高可用

### 缺点
- ❌ 冷启动延迟
- ❌ 供应商锁定
- ❌ 调试困难
- ❌ 执行时间限制
- ❌ 状态管理复杂

### 适用场景
- 不规则流量的应用
- 事件驱动任务
- 初创公司MVP
- 后台任务处理
- API端点

### 示例
```
API Gateway → Lambda Functions
              ├─ userHandler.js
              ├─ orderHandler.js
              └─ paymentHandler.js
                       ↓
              [DynamoDB / S3 / RDS]
```

## 五、六边形架构 (Hexagonal Architecture) / 端口和适配器

### 概述
也称为"端口和适配器"架构,将业务逻辑与外部依赖隔离。

### 核心概念
- **核心领域**: 业务逻辑,不依赖外部
- **端口 (Ports)**: 定义与外部交互的接口
- **适配器 (Adapters)**: 实现端口,连接外部系统

### 结构
```
         Adapters              Core              Adapters
    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
    │   REST API   │───▶│              │◀───│   Database   │
    └──────────────┘    │   Business   │    └──────────────┘
    ┌──────────────┐    │     Logic    │    ┌──────────────┐
    │   GraphQL    │───▶│              │◀───│  External    │
    └──────────────┘    │              │    │     API      │
    ┌──────────────┐    │              │    └──────────────┘
    │     CLI      │───▶│              │◀───┌──────────────┐
    └──────────────┘    └──────────────┘    │  Message     │
                                             │    Queue     │
                                             └──────────────┘
```

### 优点
- ✅ 业务逻辑完全独立
- ✅ 易于测试
- ✅ 灵活替换外部依赖
- ✅ 清晰的依赖方向

### 缺点
- ❌ 初始设置复杂
- ❌ 学习曲线陡峭
- ❌ 可能过度设计

### 适用场景
- 领域驱动设计(DDD)
- 需要高度可测试性
- 外部依赖多且可能变化

## 六、CQRS (Command Query Responsibility Segregation)

### 概述
将读操作(查询)和写操作(命令)分离到不同的模型。

### 核心思想
- **命令模型**: 处理写入,关注业务规则
- **查询模型**: 处理读取,关注展示需求
- 读写使用不同的数据模型和存储

### 优点
- ✅ 读写独立优化
- ✅ 可扩展性强
- ✅ 性能优化
- ✅ 职责清晰

### 缺点
- ❌ 复杂度增加
- ❌ 最终一致性
- ❌ 需要同步机制

### 适用场景
- 读写负载差异大
- 复杂查询需求
- 高性能要求
- 结合事件溯源

### 示例
```
       Commands                      Queries
          │                             │
    ┌─────▼──────┐               ┌─────▼──────┐
    │Command     │    Events     │Query       │
    │Handler     │──────────────▶│Handler     │
    └─────┬──────┘               └─────┬──────┘
          │                             │
    ┌─────▼──────┐               ┌─────▼──────┐
    │Write DB    │               │Read DB     │
    │(Normalized)│               │(Denormalized)│
    └────────────┘               └────────────┘
```

## 七、服务网格 (Service Mesh)

### 概述
为微服务间通信提供基础设施层,处理服务发现、负载均衡、故障恢复、监控等。

### 代表产品
- Istio
- Linkerd
- Consul Connect

### 核心功能
- 服务发现
- 负载均衡
- 故障恢复(重试、超时、熔断)
- 安全通信(mTLS)
- 可观测性(监控、追踪)
- 流量管理(金丝雀发布、蓝绿部署)

### 架构
```
Service A                Service B
    │                        │
┌───▼────┐              ┌───▼────┐
│Sidecar │◀────────────▶│Sidecar │
│Proxy   │              │Proxy   │
└────────┘              └────────┘
    │                        │
    └────────▼───────────────┘
         Control Plane
    (Policy, Config, Telemetry)
```

### 优点
- ✅ 与业务逻辑解耦
- ✅ 统一的服务治理
- ✅ 安全加固
- ✅ 可观测性增强

### 缺点
- ❌ 复杂度增加
- ❌ 性能开销
- ❌ 学习曲线陡峭

### 适用场景
- 大规模微服务
- 需要细粒度流量控制
- 安全要求高
- 多语言服务

## 八、领域驱动设计 (DDD)

### 概述
一种软件开发方法论,强调领域模型和业务逻辑。

### 核心概念

#### 战略设计
- **限界上下文 (Bounded Context)**: 明确的边界
- **上下文映射 (Context Map)**: 上下文间关系
- **子域 (Subdomain)**: 核心域、支撑域、通用域

#### 战术设计
- **实体 (Entity)**: 有唯一标识的对象
- **值对象 (Value Object)**: 无标识,不可变
- **聚合 (Aggregate)**: 一组相关对象
- **聚合根 (Aggregate Root)**: 聚合的入口
- **领域服务 (Domain Service)**: 跨实体的业务逻辑
- **仓储 (Repository)**: 持久化接口
- **工厂 (Factory)**: 创建复杂对象

### 优点
- ✅ 业务语言一致
- ✅ 复杂业务建模清晰
- ✅ 高内聚低耦合
- ✅ 易于演进

### 缺点
- ❌ 学习曲线陡峭
- ❌ 初期投入大
- ❌ 可能过度设计

### 适用场景
- 复杂业务逻辑
- 长期项目
- 大型团队
- 业务快速变化

## 九、云原生架构

### 核心原则
1. **容器化**: Docker, Kubernetes
2. **微服务**: 服务独立
3. **动态编排**: K8s自动管理
4. **DevOps**: CI/CD自动化
5. **可观测性**: 监控、日志、追踪

### 12-Factor App
1. 基准代码 - 版本控制
2. 依赖管理 - 显式声明
3. 配置外部化 - 环境变量
4. 后端服务 - 附加资源
5. 构建、发布、运行 - 严格分离
6. 进程无状态 - 不保存本地状态
7. 端口绑定 - 服务导出
8. 并发 - 水平扩展
9. 快速启动和优雅终止
10. 开发/生产环境一致
11. 日志流处理
12. 管理进程 - 一次性任务

### 优点
- ✅ 可移植性
- ✅ 弹性伸缩
- ✅ 快速部署
- ✅ 容错性强

### 适用场景
- 云端应用
- 需要频繁部署
- 流量波动大
- 多环境部署

## 十、架构模式选择矩阵

| 需求/场景 | 推荐架构 | 原因 |
|----------|---------|------|
| 小团队,快速迭代 | 分层架构 / 模块化单体 | 简单,易上手 |
| 大型企业应用 | 微服务 | 团队自治,独立部署 |
| 实时数据处理 | 事件驱动 | 异步,高吞吐 |
| 不规则流量 | Serverless | 成本优化,自动扩展 |
| 复杂业务逻辑 | DDD + 六边形架构 | 清晰建模,易维护 |
| 读写分离优化 | CQRS | 性能优化 |
| 大规模微服务治理 | 服务网格 | 统一治理 |
| 云端应用 | 云原生架构 | 充分利用云优势 |

## 十一、架构演进路径

### 典型演进
```
单体应用
    ↓ (团队扩大,业务复杂)
模块化单体
    ↓ (需要独立部署)
微服务
    ↓ (通信复杂)
事件驱动微服务
    ↓ (流量不规则)
Serverless微服务
    ↓ (大规模治理)
服务网格
```

### 渐进式迁移
1. 识别边界 - 找到服务边界
2. 提取服务 - 逐步拆分
3. 数据迁移 - 数据库分离
4. 切换流量 - 灰度发布
5. 优化监控 - 完善可观测性

### 关键原则
- 不要过早微服务化
- 保持单体的模块化
- 先解决组织问题
- 逐步演进,不要大爆炸
- 持续监控和优化
